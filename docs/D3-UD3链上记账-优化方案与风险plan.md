# D3/UD3 链上记账 —— 优化方案与风险 Plan（Task 5 后续）

> 承接《D3-链上链下架构与推荐绑定合约设计.md》。目标：把「质押/产出记 D3、不发 D3 代币；
> UD3 按链上推荐 S 档记账；链下拨出的 UD3 不超过链上记账」这套构想，从**信任保证**逐步升级为

## 1. 三个强度档位（先选目标强度，再排期）

| 档位 | 保证类型 | 「拨出 ≤ 记账」由谁保证 | 成本 | 适用 |
|---|---|---|---|---|
| **A. 仅锚定（evidence-only）** | 事后可审计 | 无强制，仅用户可自证自己余额 | 低（一天一笔 gas） | 现在就能上，最快 |
| **B. 封顶出款（capped-payout, 推荐）** | 出款那一刻链上校验额度 | 出款合约校验「本次 ≤ 当日锚定根中该用户额度（带 Merkle proof）」 | 中（改出款路径 + 每日发额度根） | 想要硬封顶但不发币 |
| **C. 全链上账本** | 完全 trustless | 记账/S 档/业绩全部上链或 ZK | 高（gas + 复杂度） | 只有发 ERC-20 上 DEX 才值得 |

> **建议目标 = B**。A 不能阻止超发，C 投入产出比差。B 用「锚定根 + proof 封顶」把超发从
> 「信任运营方」变成「链上不可能」，且不需要把业绩计算搬上链。

## 2. 推荐路线（阶段化，可随时停在任一阶段）

### P0 · 偿付能力监控（最优先，独立于合约）
- 实时 `闪兑钱包USDT储备 / 全网待兑付负债` 比值；低于阈值告警 + 从金库动态补给。
- 待兑付负债口径：**只算「已释放未闪兑的 D3 折 USDT」**（UD3 本身不可提现，不是即时负债）。
- 风险最高点，先做。落点：一个 cron + 告警，不动合约。

### P1 · 锚定上线（= 档位 A）
- 每日结算后算 `{wallet → {d3记账, ud3记账}}` 的 Merkle 树，`DailyStateAnchor.anchor(dateKey, root, leafCount)`。
- 前端/后端提供「查询我的 proof + 自证余额」入口（合约已有 `verify()`）。
- `leafCount` 公示，配合叶子集合闭包声明，压缩「选择性遗漏」空间。

### P2 · 额度根 + 封顶出款（= 档位 B 核心）
- 每日额外锚定一棵「**UD3 可支配额度根**」：叶子 = `keccak(wallet, cumulativeUd3Entitlement)`，
  `entitlement` = 按链上 S 档 + 业绩算出的**累计**上限。
- 新出款/闪兑合约在出款时要求调用方提交 `(wallet, amount, cumulativeSpent, proof)`：
  - 校验 `proof` 对当日额度根成立；
  - 校验 `cumulativeSpent + amount ≤ entitlement`；
  - 合约内累计 `spent[wallet] += amount`，防重复花。
- 出款资金仍由 Turnkey 多签充值该合约；**多签不能绕过合约超付**（合约是唯一出款口）。

### P3 · 计价单一价源 → 预言机可切换
- `d3Price.ts` 单一价源；管理价改动走多签 + 审计日志；未来切 TWAP 预言机零改业务。

## 3. 各阶段风险 + 缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| **偿付不足（挤兑）** | 最易爆；闪兑付不出 USDT | P0 储备/负债监控 + 阈值补给；UD3 不可提现天然压负债 |
| **选择性不锚 / 锚假根遗漏超发** | A 档下运营方仍可超发后不锚 | 升到 B 档才根治；A 档辅以 leafCount 公示 + 用户端主动验证告警 |
| **额度根算错/被操纵（B档）** | 链上会照假额度放款 | 额度计算确定性可复算 + 多签复核根 + 根写一次性；争议期（如 24h）后才可对新根出款 |
| **Merkle proof 重放 / 跨日** | 用同一 proof 多次取 | 合约内 `spent[wallet]` 累计 + 额度用**累计上限**语义（非单日增量） |
| **D3 管理价被误改/内部操纵** | 放大兑付 | 改价多签 + 审计日志 + 变动幅度限速；P3 切预言机 |
| **S 档链上、比例/业绩链下** | 「链上记账」可信度=链下算法 | B 档把「额度」锚成根即可封顶；无需把业绩上链 |
| **anchorer/多签私钥泄露** | 伪造根 / 越权出款 | Turnkey 多签 + 角色最小化 + 根争议期 + 监控异常出款事件 |
| **BSC reorg/最终性** | 锚定/出款回滚 | 出款等 N 确认；锚定按已终结区块 |

## 4. 落地检查清单（做 B 档时逐条打勾）

- [ ] P0 偿付监控上线并告警联通
- [ ] 每日 Merkle（余额根 + UD3 额度根）计算脚本，确定性可复算
- [ ] `DailyStateAnchor` 部署 + anchorer 交给 Turnkey 多签
- [ ] 出款合约：proof 校验 + `spent` 累计 + 唯一出款口 + 根争议期
- [ ] 出款资金只经合约；关闭一切「绕过合约」的直付路径
- [ ] 前端：用户自证余额 / 额度入口
- [ ] 回归测试：超付被拒、重放被拒、跨日额度正确、reorg 安全

## 5. 一句话结论

**先做 P0（偿付监控）+ P1（锚定），把「可审计」拿到手；想要「拨出不可能超发」的硬保证，
再上 P2 的「额度根 + 封顶出款合约」——这是不发 ERC-20 前提下投入产出比最高的一步。
档位 C（全链上账本）留给未来发币上 DEX 时再谈。**
