四、【本次新发现】金库资金管理 / 转账流(T 系列)

架构:后台管理员 → submitTreasuryTransfer 构造 BSC 交易(BNB/USDT)→ 提交 Turnkey SIGN_TRANSACTION → 因金库钱包(应)受 2/3 root-quorum 保护而返回 CONSENSUS_NEEDED,由 3 名 root 中 2 名在 Turnkey 面板批准后广播。签名层设计正确,但应用层缺乏独立控制,"非单人可掏空"完全依赖链下 Turnkey 配置,代码不强制。

ID发现严重度证据 / 修复
T-Cdev_hd 内联签名+即时广播绕过全部 consensus:若金库钱包被 dev 方式provision 或 2/3 阈值未提升,单一 members.write 管理员一次调用即可把金库全额转到任意地址Critical(配置相关)turnkey.ts:549、fundManagement.ts:155。修复:生产强制拒绝金库 dev_hd;代码断言/校验 Turnkey 阈值≥2
T-A金库出账仅由低阶共享权限 members.write 门控(与普通会员管理/生成充值地址同权);无专用财务/金库权限;superadmin 全绕过Highadmin/index.ts:1066,1101。修复:新增专用 treasury/finance 权限或仅 superadmin
T-B金库转账无应用层 maker-checker:同一管理员可 propose 又 broadcast;assertDifferentApprover 未接入金库流(其余高危面都有)Highadmin/index.ts 未走 createApproval。修复:propose→broadcast 走双人审批
T-D无金额上限 / 收款白名单 / 每日限额;且 turnkey-setup-policies.ts:72 注释谎称"应用层已有单笔/每日限额"——实际不存在Med-HighfundManagement.ts:118。修复:服务端加最大额+白名单+日限;删除误导注释
T-E无每笔幂等键:两次相同 propose → 两条记录 → 两个 Turnkey 活动 → 若都被批准则两笔链上转账Med迁移 047 无幂等键。修复:加 request idempotency key 唯一约束
T-JTurnkey 策略兜底 DENY(condition:'true') 与"金库应 fall-through 到 root-quorum"的 DSL 语义未在仓内验证Med(运维/正确性)链上实测确认金库转账确实到达 CONSENSUS_NEEDED 而非被直接拒
—正面:to/amount 编入 Turnkey 签名载荷、广播不重读 DB(无 TOCTOU);047 有 amount>0/asset/status CHECK + RLS anon 撤销;propose/broadcast 均写审计GOOD—
净新增风险:一条金库出账路径,唯一实质保障是链下 Turnkey 2/3 consensus;应用层无 maker-checker、低阶共享权限、无额度/白名单/幂等,并有 dev_hd 单人掏空旁路。建议在承载真实资金前按上表修复,并链上核实金库确为真实 2/3 Turnkey 钱包。
